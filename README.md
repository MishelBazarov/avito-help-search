# Поиск статей справки по вопросу пользователя

Тестовое задание Avito DS Bootcamp, направление NLP & LLM.

По тексту обращения пользователя нужно вернуть ранжированный список `article_id`, которые дальше уйдут в RAG-пайплайн. Метрика `MAP@10`.

**Качество:** на `calibration.f` (5-fold cross-validation с разбиением по запросам) MAP@10 около **0.60**; на тестовой выборке платформы MAP@10 = **0.564**. Небольшой разрыв ожидаем: гиперпараметры реранкера подбирались по CV на calibration, поэтому на нём оценка чуть оптимистичнее.

## Структура

```
solution.ipynb   весь пайплайн от чтения данных до answer.csv, с графиками
answer.csv       предсказания для test.f
```

Для запуска положить `articles.f`, `calibration.f`, `test.f` рядом с ноутбуком и выполнить все ячейки. `answer.csv` пересоздаётся заново.

## Подход

Двухступенчатая схема: сначала быстрый отбор кандидатов несколькими методами поиска, потом переранжирование обучаемой моделью.

### Обработка HTML

Текст статей чищу через BeautifulSoup с предварительным `html.unescape`. Таблицы, вкладки и спойлеры в этой разметке лежат статически, поэтому `get_text()` достаёт и их содержимое, отдельная обработка не нужна.

Дальше токенизация по кириллице, латинице и цифрам, затем лемматизация через pymorphy3. Лемматизация здесь обязательна: русский язык флективный, и без неё "отправить", "отправляю" и "отправка" не совпадут между запросом и статьёй. Словари идут внутри пакета, в сеть решение не ходит.

Отдельно вырезаю плейсхолдеры анонимизации (`<DATE>`, `<MONEY>`, `<PHONE>`, `<ID>`, `<URL>`). Они встречаются примерно в 9% запросов, и если их оставить, лемматизатор разбирает их как английские слова, что шумит в лексических признаках.

### Разбиение статей на пассажи

Правильный ответ часто это статья-хаб с общим заголовком вроде "Покупателю" или "Продавцу", внутри десяток разных подтем, а нужный фрагмент занимает пару предложений в длинном тексте. В скоре по всему документу такой фрагмент растворяется.

Поэтому статьи длиннее 150 слов режу на пересекающиеся окна по 150 слов с шагом 100. Перекрытие нужно, чтобы фраза не разрывалась ровно по границе окна. Заголовок приписываю каждому чанку с тройным весом, а при поиске беру максимум скора по чанкам одной статьи. 793 статьи превращаются в 5734 чанка. По замеру это поднимает MAP гибрида с 0.40 (по документам) до 0.42 (по чанкам).

### Сигналы поиска

Основа лексическая, четыре сигнала закрывают разные типы промахов:

- **BM25 по леммам** - точное совпадение слов с учётом морфологии.
- **TF-IDF по символьным n-граммам (3-5)** - добирает опечатки и формы, которые лемматизатор не свёл к словарной. Например, "Москлвской" не приводится к "московский", но пересекается с ним по большинству триграмм.
- **TF-IDF по словесным n-граммам (1-2) по леммам** - ловит устойчивые словосочетания ("пункт выдачи", "статус заказа").
- **LSA (TruncatedSVD, 200 компонент)** поверх словесной TF-IDF статей - латентно-семантический сигнал, сближает термины, которые часто встречаются в одних статьях. На корпусе из 793 документов это слабый сигнал: по отдельности как ретривер почти не работает, беру его только как признак реранкера.

Взвешенный гибрид BM25 + char даёт 0.42 и служит источником кандидатов. Остальные сигналы по отдельности слабее, но как признаки реранкера дают вклад.

В ноутбуке есть опциональный хук для локальных нейро-эмбеддингов (`USE_EMB`, по умолчанию выключен). Он легален по правилам (локальная open-source модель уровня MiniLM/e5, ~100-300M параметров, без внешних API), но в приложенный `answer.csv` не входит: не хватило времени и ресурсов на полный прогон и тюнинг с ними. Это направление на развитие, а не ограничение задания.

### Реранкер

Кандидаты: объединение top-30 от BM25 и char-TF-IDF. По замеру top-30 покрывает 94.6% релевантных статей; больше 30 не помогает: recall почти не растёт, а реранкер получает больше шума. Word-TF-IDF и LSA в отбор кандидатов не добавляю, они идут только как признаки.

Признаки на пару запрос-кандидат:

- скор, ранг и разрыв со скором лидера выдачи для каждого сигнала;
- BM25 отдельно по заголовку и отдельно по телу статьи;
- пересечение лемм запроса с заголовком и со всей статьёй, в штуках и в долях от длины запроса;
- длины заголовка и статьи.

Модель: `HistGradientBoostingClassifier` из sklearn, предсказывает вероятность релевантности, по ней сортирую. Гиперпараметры (`max_iter=500`, `max_leaf_nodes=15`, `learning_rate=0.05`, `l2_regularization=1.0`) подобраны перебором по CV; в коде рядом оставлен вариант с ранней остановкой, если захочется уменьшить оптимистичность CV.

Отдельно проверял IDF-взвешенные признаки пересечения (совпадение по редкому слову информативнее, чем по частому). На CV они дали нулевой вклад: с ними и без них MAP совпадает в пределах шума. К тому же сумма IDF по множеству лемм вносила зависимость от `PYTHONHASHSEED` и ломала воспроизводимость. Поэтому в финальный набор они не вошли, их роль берут на себя скоры BM25 и TF-IDF.

## Как проверял качество

5-fold cross-validation на `calibration.f` с разбиением **по запросам**, а не по строкам кандидатов. Это принципиально: у одного запроса 40-50 строк-кандидатов, и если бить выборку по строкам, кандидаты одного запроса попадут и в train, и в validation, а оценка окажется завышенной. Поскольку разброс между фолдами около 0.02-0.03, а разница между конфигурациями бывает меньше, проверяю на нескольких сидах разбиения, чтобы не принять шум за прирост.

Прогресс по шагам:

| Вариант | MAP@10 (5-fold CV) |
|-|-|
| гибрид по документам | 0.40 |
| гибрид по чанкам (baseline) | 0.42 |
| + реранкер на BM25 + char | 0.59 |
| + word-TF-IDF и LSA | 0.60 |

Все числа печатаются ячейками ноутбука и продублированы графиками (вклад отдельных сигналов, MAP по этапам, кривая recall кандидатов, распределение AP@10 по запросам до и после реранкера). Нейро-эмбеддинги в приложенном результате отключены (`USE_EMB=False`).

Финальную модель обучаю на всех 500 размеченных запросах и применяю к `test.f`.

## Какие ошибки нашёл при анализе

**Статьи-хабы с общими заголовками.** Основная причина полных промахов: заголовок правильной статьи слишком общий, чтобы поймать её лексически. Решено разбиением на пассажи с агрегацией по максимуму.

**Плейсхолдеры анонимизации.** Лемматизировались как обычные слова и добавляли ложные совпадения между запросами. Вырезаю до токенизации.

**Опечатки и разговорные формы.** Лемматизатор нормализует не всё. Компенсируется символьными n-граммами, они устойчивы к искажениям независимо от морфологии.

**Семантически близкие, но лексически разные пары.** Пример из калибровки: "Авито теперь не работает за наличку??" при правильном ответе "Оплата заказов с доставкой", общих слов почти нет. Частично добирается LSA; полноценно закрыл бы нейросетевой энкодер (см. развитие).

## Воспроизводимость

Пайплайн детерминирован: два независимых прогона дают побайтово одинаковый `answer.csv`. Все сиды зафиксированы (`SEED=42` для модели и разбиений CV, `random_state` у TruncatedSVD), а признаки не зависят от порядка обхода множеств.

## Как включить нейро-эмбеддинги (эксперимент)

```
pip install sentence-transformers
```

Поставить `USE_EMB=True` в первой ячейке и выполнить ноутбук: ячейка подхватит локальную модель, добавит эмбеддинги как пятый сигнал и пересчитает `answer.csv`. Первый запуск скачает веса модели (локально, один раз).

## Ограничения и что дало бы ещё прирост

По убыванию ожидаемой отдачи:

1. **Ранжирующий лосс.** Метрика ранжирующая (MAP@10), а реранкер учится бинарной классификацией. `LGBMRanker` с `objective="lambdarank"` (LambdaMART) оптимизирует ранжирование напрямую - самый вероятный реальный прирост. LightGBM ставится локально (`pip install lightgbm`), внешних сервисов не требует.
2. **Нейро-эмбеддинги.** Локальный энкодер (<1B параметров, без внешних API, по правилам легально) добирает семантические промахи, где смысл совпадает, а слова разные. В текущем решении не задействованы из-за ограничений по времени и ресурсам, а не по правилам.
3. **Словарь синонимов предметной области** (доставка/отправка, возврат/отмена) - дёшево и бьёт ровно в лексические промахи.
4. **Нарезка по границам HTML-блоков** вместо равномерных окон.

LSA держу со скромными ожиданиями: латентная семантика любит корпуса покрупнее, на 793 статьях как отдельный ретривер вряд ли выстрелит, полезнее как признак.

## Стек

Python, pandas, numpy, BeautifulSoup, lxml, pymorphy3, rank_bm25, scikit-learn. Опционально: matplotlib для графиков, sentence-transformers + torch для нейро-эмбеддингов, lightgbm для ранжирующего реранкера.

Внешние LLM и платные API не используются нигде, включая инференс на тестовой части. Все модели и словари работают локально.
